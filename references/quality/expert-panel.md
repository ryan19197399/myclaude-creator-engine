# Expert Panel — Stage 7 Validation Protocol

**Purpose:** Simulate domain experts debating product quality before publish.
**When:** MCS-2+ validation, after Stages 1-6 pass.
**Effect:** Scoring (not just coaching). Products below threshold get specific remediation.

---

## Panel Composition

For each product, simulate 3 expert perspectives:

| Expert | Perspective | What They Challenge |
|--------|------------|-------------------|
| **Domain Specialist** | "Does this encode real expertise?" | Depth of knowledge, proprietary methodology, domain-specific patterns |
| **Buyer Advocate** | "Would I pay for this?" | Value proposition, clarity, install-and-use experience, time saved |
| **Platform Architect** | "Does this strengthen the ecosystem?" | Composability, uniqueness in catalog, technical quality |

---

## Scoring Protocol

Each expert scores independently on their dimension (0-100):

### Domain Specialist Score (0-40)

| Score | Criteria |
|-------|---------|
| 35-40 | Product encodes proprietary methodology not available elsewhere. references/ has deep domain knowledge. Creator's unique expertise is structurally embedded. |
| 25-34 | Solid domain knowledge. Some unique perspective but primary approach is well-known. |
| 15-24 | Generic domain knowledge. Could be generated by AI without domain expert input. |
| 0-14 | No domain expertise visible. Template-level content. |

**Detection method:** Read references/ directory. Count domain-specific terms not in common AI vocabulary. Check if routing/decision trees reflect real-world complexity.

### Buyer Advocate Score (0-30)

| Score | Criteria |
|-------|---------|
| 25-30 | Clear value proposition. Solves specific problem. Install → value in <5 minutes. README makes buyer confident. |
| 18-24 | Good value but unclear differentiation from free alternatives. |
| 10-17 | Vague value proposition. Buyer unsure what they get. |
| 0-9 | No clear buyer value. |

**Detection method:** Read README. Time from install to first useful output. Specificity of use cases. Presence of "before/after" examples.

### Platform Architect Score (0-30)

| Score | Criteria |
|-------|---------|
| 25-30 | Composable with other products. Fills unique catalog gap. Uses CC platform features well (hooks, memory, context). |
| 18-24 | Works well standalone but doesn't leverage platform. Minor catalog overlap. |
| 10-17 | Significant overlap with 3+ existing products. Limited platform integration. |
| 0-9 | Duplicate of existing product. No platform awareness. |

**Detection method:** Search marketplace catalog for similar products (grep product descriptions). Check frontmatter for platform features (context, hooks, memory, paths).

---

## Composite Score

```
EXPERT_PANEL_SCORE = domain_score + buyer_score + architect_score
```

| Total | Verdict | Action |
|-------|---------|--------|
| >= 70 | **DISTINGUISHED** | Proceed to publish. Product is differentiated. |
| 50-69 | **ADEQUATE** | Proceed with specific improvement suggestions. |
| 30-49 | **NEEDS DIFFERENTIATION** | Strong warning + remediation plan. Creator can override with explicit confirmation (CE-D26: coaching, not gatekeeping). |
| < 30 | **COMMODITY** | Strong warning + major rework guidance. Creator can still publish with explicit override but receives marketplace visibility penalty notice. |

---

## Remediation Template

When score < 50:

```
EXPERT PANEL REVIEW — {product_name}

Panel Score: {total}/100
  Domain Specialist: {domain}/40 — {one-line finding}
  Buyer Advocate: {buyer}/30 — {one-line finding}
  Platform Architect: {platform}/30 — {one-line finding}

Verdict: {DISTINGUISHED|ADEQUATE|NEEDS DIFFERENTIATION|COMMODITY}

Top 3 improvements to raise score:
1. {specific action targeting lowest-scoring dimension}
2. {specific action targeting second-lowest dimension}
3. {specific action based on marketplace gap analysis}

Expert panel review is required for MCS-2+ certification.
Products at MCS-1 skip this review (minimum viable quality only).
```

## Integration with /validate

Stage 7 in /validate should:
1. Load this file: `references/quality/expert-panel.md`
2. Load anti-commodity spec: `references/quality/anti-commodity.md` (for marketplace comparison)
3. Simulate each expert perspective by reading the product files
4. Compute composite score
5. Report verdict with remediation if needed
6. For MCS-2: scores < 30 are BLOCKING. Scores 30-49 are WARNING.
7. For MCS-3: scores < 50 are BLOCKING. Scores 50-69 are WARNING.

This replaces the coaching-only approach with a scored, evidence-based review.
