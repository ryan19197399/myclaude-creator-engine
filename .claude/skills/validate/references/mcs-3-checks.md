# MCS-3 Checks — State-of-the-Art

The gold standard. Products that set the bar for the entire marketplace. Requires agent review.

**Prerequisite:** All MCS-1 and MCS-2 checks must pass before MCS-3 checks are evaluated.

**Total additional checks in this tier:** 9 (all agent-assisted)

---

## Additional Requirements (beyond MCS-2)

| # | Check | Description | Type |
|---|-------|-------------|------|
| E1 | Deep knowledge base | `references/` directory contains real domain expertise (not generic content) | Agent review |
| E2 | Adaptive modes | Product works in different contexts or under different constraints | Agent review |
| E3 | Composability | Product integrates well with other standard MyClaude products | Agent review |
| E4 | Stress-tested | Handles ambiguity, adversarial input, and edge cases gracefully | Agent review |
| E5 | Cognitive architecture documented | WHY the product is designed this way is explicitly documented | Agent review |
| E6 | Versioning strategy | Changelog exists and versioning strategy is documented | Agent review |
| E7 | Exemplars — edge case coverage | At least 5 exemplars, and at least 2 cover edge cases or non-obvious use cases | Agent review |
| E8 | Token efficiency | Product does not waste context — instructions are dense and purposeful | Agent review |
| E9 | Differentiation statement | Explicit statement of what makes this product unique and why AI alone couldn't produce it | Agent review |

## Agent Review Criteria

### E1 — Depth Review

The reviewing agent reads `references/` and evaluates:
- Does the content reflect deep domain expertise or is it generic?
- Could this content have been generated by a default LLM without domain knowledge?
- Does it encode specific frameworks, methodologies, or data that a practitioner would recognize?

**PASS:** References demonstrate expert-level domain knowledge specific to the product's use case.
**FAIL:** References contain only general information available in any textbook or basic web search.

### E2 — Adaptive Modes

The reviewing agent checks whether the product:
- Explicitly documents how it behaves under different constraints (time pressure, limited context, different user expertise levels)
- Has variant configurations or modes
- Gracefully degrades when full resources aren't available

**PASS:** At least 2 distinct operational modes or contexts are documented and functional.
**FAIL:** Product assumes ideal conditions and fails or behaves unpredictably in variations.

### E3 — Composability Test

The reviewing agent evaluates whether the product:
- Can be invoked from or by other standard MyClaude products (skills, workflows, agents)
- Has clear input/output contracts that other products can depend on
- Does not have hidden dependencies or hard-coded assumptions that break composition

**PASS:** Product has clean interfaces and documented integration points.
**FAIL:** Product is tightly coupled to a specific context or has undocumented assumptions.

### E4 — Stress Test

The reviewing agent generates and evaluates 3 test cases:
1. **Ambiguity test:** underspecified input with multiple valid interpretations
2. **Edge case test:** valid but unusual input that exercises boundary conditions
3. **Adversarial test:** input designed to confuse or exploit the product

**PASS:** Product handles all 3 cases gracefully — either produces good output or fails with a clear, helpful error.
**FAIL:** Product produces incorrect output, hallucinates, loops, or fails silently on any test case.

### E5 — Architecture Review

The reviewing agent checks whether the product:
- Documents WHY key design decisions were made (not just WHAT the design is)
- Explains the cognitive model or mental framework underlying the product
- Anticipates common evolution paths and explains why alternatives were rejected

**PASS:** A new creator could read the architecture notes and understand why the product works the way it does.
**FAIL:** Design decisions are implicit or undocumented — the product is a black box.

### E6 — Versioning Strategy

The reviewing agent verifies:
- `CHANGELOG.md` exists with at least the initial version entry
- Versioning strategy is documented (what triggers a MAJOR vs. MINOR vs. PATCH bump)
- Current version in all files is consistent

**PASS:** Changelog present, versioning strategy documented, version consistent across files.
**FAIL:** No changelog, no version strategy, or inconsistent versions.

### E7 — Exemplar Coverage

The reviewing agent reads all exemplars and classifies them:
- Happy path (clear, straightforward use case)
- Edge case (unusual but valid use case)
- Boundary case (testing limits of the product)
- Negative case (what the product handles by refusing or redirecting)

**PASS:** At least 5 total exemplars, with at least 2 classified as edge, boundary, or negative cases.
**FAIL:** Fewer than 5 exemplars, or all exemplars are happy-path only.

### E8 — Token Efficiency

The reviewing agent evaluates context usage:
- Are instructions dense and purposeful, or padded with repetition?
- Are there sections that could be removed without loss of functionality?
- Does the product avoid loading unnecessary context for common operations?

**PASS:** Instructions are precise. Token usage is proportional to the problem complexity.
**FAIL:** Significant redundancy, over-specification of obvious things, or loading large references for every invocation.

### E9 — Differentiation Statement

The reviewing agent looks for an explicit section or statement that answers:
1. What domain expertise went into this that AI alone couldn't generate?
2. What remains if AI-generated content is removed?
3. What specific problem (<5 other products address) does this solve?

**PASS:** All three questions are answered explicitly and credibly in the product documentation.
**FAIL:** No differentiation statement, or the answers are generic ("this is high quality", "uses best practices").

## MCS-3 Scoring

MCS-3 score is calculated across all tiers:
- MCS-1 checks: weighted 20%
- MCS-2 checks: weighted 30%
- MCS-3 checks: weighted 50%

A product must score 90/100 or above across all tiers to receive MCS-3 certification.
All E-level checks are REQUIRED — no partial credit for failing agent checks.
